# 从十万上下文到十亿总 Token：基于 TokenMart 的单任务超大规模调度架构初论

## 摘要

本文讨论一个看似矛盾、实则可以被架构化解的问题：如果单个智能体会话通常只能稳定使用约十万 token 级别的上下文，那么单个任务是否仍可能在总体上调度到十亿 token，甚至进一步逼近“十亿 token 级任务”的求解门槛？本文的核心论点是：答案是肯定的，但前提不是把单个会话无限拉长，而是把“任务状态、协作关系、验证义务、奖励结算与信用约束”从单次会话内部外部化，沉淀为一个可持续的任务图、通信图与结算图。

这种思路与《单任务-万亿级Token调度架构框架-工作论文》中的核心命题一致：决定能力上限的并不只是单轮上下文窗口，而是围绕同一目标可被调度的总 token 预算、其中间表示的压缩损失、以及系统把 token 转化为有效工作的“认知卡诺效率”η。与此同时，《OpenClaw的行为模式》提供了一个极其重要的反证材料：如果智能体仍停留在高提示税、高 shell 依赖、高输出洪泛、低交付闭环的会话形态，那么它就是一种典型的有损代理，无法稳定承担长程、重组合、强验证的超大规模单任务执行。

本文据此提出：TokenMart 的价值不在于它已经自动解决了十亿级任务，而在于它给出了一套更接近可扩展答案的制度化骨架。它把单任务包装为 `mountain -> campaign -> work_spec -> work_lease -> deliverable -> verification_run -> replan -> reward` 的多层运行时结构，把 TokenBook 作为协作与共享记忆层，把 TokenHall 作为推理与结算共用的信用轨道，把信任拆分为 `service_health`、`market_trust` 与 `orchestration_capability` 三个不同维度。正是在这种“任务分解、代理通信、验证、激励、结算”被统一约束的情况下，许多局部只有十万 token 工作带宽的代理，才有可能共同参与一个总体消耗达到十亿 token 的单一任务。

## 一、问题重述：单会话上限，不等于单任务上限

若我们接受一个现实前提，即绝大多数代理会话都不能长期可靠地持有超大上下文，那么单任务扩展就不能再依赖“一个越来越长的会话”。它必须依赖另外三件事：

1. 把长期状态外部存储化。
2. 把任务拆成可验证的中间表示。
3. 把不同局部代理之间的交接损失压低到可接受范围。

第一篇附件已经给出最关键的理论支点。它将 token 量级视为“认知能量”，并指出能力扩展不是线性的，而是存在相变；更重要的是，它证明了只要存在外部存储，任何超过单次上下文窗口的可计算任务，在原则上都可以被分页式地拆解执行。真正的难点不在“能不能拆”，而在“拆了之后损失多大”。该文把这个问题刻画为状态压缩损失 epsilon 的问题：如果跨步骤传递的中间状态表示得不好，质量会随分解次数指数级衰减。换言之，超大规模单任务不是先天不可能，而是高度依赖架构。

这也解释了为什么“单会话只有十万 token”并不能直接推出“任务最多只有十万 token 级”。会话窗口只是工作带；只要任务真相不依赖于某个单会话内部的完整保留，而是能够被不断外化为结构化工件、验证记录、交接契约和信用状态，那么总体 token 消耗可以远超任一局部代理的窗口上限。

## 二、OpenClaw 反例：为什么单体会话代理天然是有损的

第二篇附件提供了本文必须面对的现实约束。它不是在证明代理不能做事，而是在证明：如果我们把大规模任务寄托在 OpenClaw 这类会话式代理自身的“连续聪明”上，那么损失会非常高。

这份研究最重要的发现有三类。

第一，它是高热损失的。文中统计 48 个 SWE-bench finished runs 后发现，在 47 个可读 prompt report 中，系统提示、技能提示、工具 schema 和注入文件构成的固定开销平均达到 56,982 chars，而项目上下文平均只有 12,802 chars，固定税约占 81.65%。工具调用又以 `exec` 为主，占 61.9%，而 `edit` 与 `write` 合计仅 11.6%。这意味着大量 token 被花在系统外壳、shell 观察与上下文噪声上，而不是压缩进“诊断-修改-验证-交付”的高价值闭环里。

第二，它是中间态化的。该文指出，48 个 run 中有 11 个以 `toolUse` 结束，2 个以 `error` 结束，5 个 `patch.diff` 为零字节，且若干 run 直接停在“让我再看看测试文件”一类中间动作上。也就是说，很多会话并不是停止在“完成”，而是停止在“还在做”。对于一个超大规模单任务系统而言，这种停止方式是灾难性的，因为中间态如果没有被外部系统强制沉淀，就等于丢失。

第三，它是部分记忆化的。仓库中的 OpenClaw 研究材料表明，其上下文会被压缩，会话会按日或闲置重置，子代理上下文隔离且自动归档，长期记忆依赖按需加载和语义检索而非常驻全量记忆。这不是缺陷，而是会话式代理的常态：它们本来就应该被当作“临时工人”，而不是“持续持有整个任务真相的神经中枢”。

因此，OpenClaw 的真正启示不是“代理不行”，而是“把任务真相留在代理脑内这件事不行”。单体会话代理是有损代理、瞬时代理、部分记忆代理。它适合作为局部工作节点，不适合作为超长程单任务的唯一状态容器。

## 三、TokenMart 的方法：把任务真相从会话内部迁移到制度化外部结构

TokenMart 代码库给出的，不是一个更长上下文的代理，而是一整套把任务状态外部化的运行时方法。

首先，它给单任务提供了真正的顶层容器。首页和文档页反复把 `mountain` 叙述为“顶层 mission umbrella”，即由管理员定义目标问题、成功标准、预算包络与治理策略的单任务总框架。此后，一个 mountain 被拆成多个 `campaign`，再拆成多个 `work_spec`。每个 `work_spec` 不是一句模糊任务描述，而是自带 `input_contract`、`output_contract`、`verification_contract`、`reward_envelope`、`duplication_policy`、`checkpoint_cadence_minutes` 与依赖边。这里最关键的不是命名，而是中间表示被做实了。

其次，它把执行做成了租约制而不是聊天制。`work_lease` 明确记录谁被分配、何时接受、何时到期、何时 checkpoint 到期、何时提交、何时验证，以及失败原因与 checkpoint 载荷。对大规模单任务而言，这一点至关重要：会话结束并不等于状态结束，代理掉线也不等于任务真相消失。任务真相被保存在租约、交付物和验证记录里，而不是保存在“代理刚才脑子里还记得什么”上。

再次，它把验证与重规划前置为一等公民。`deliverable`、`verification_run` 与 `replan` 不是附属模块，而是核心运行时表。奖励也不是单一支付，而是与 `reward_role` 和 `settlement_policy_mode` 绑定，允许 proposer、executor、reviewer、synthesizer、verifier、coalition、supervisor bonus 等不同角色分润，并允许 `dynamic_difficulty`、`coalition_formula`、`replication_bonus`、`contradiction_resolution` 等不同结算模式。这意味着 TokenMart 不是只在“把任务切小”，而是在“把多类型贡献都纳入可结算的统一合约”。

最后，它把方法论写进了图结构本身。`docs/ORCHESTRATION_METHODOLOGY.md` 与网页源码中的方法论页面都要求每个节点具备 `input_spec`、`output_spec`、`passing_spec`、`verification_method`、`retry_policy`、预算和升级路径。`src/lib/orchestration/plans.ts` 又进一步把分解质量量化为 `decomposition_coverage`、`reviewer_agreement_rate`、`rework_rate`、`handoff_success_rate`、`forecast_accuracy`、`duplicate_work_avoidance`、`evidence_density` 等指标。更关键的是，plan review 不是单人自证，而是明确分成 planner、reviewer、reconciler 三阶段，且存在先后门槛与行为隔离。换言之，系统不是事后才回顾“这个协作做得好不好”，而是在执行层就把“低损耗交接”和“多方独立确认”写成评分对象。

这正是第一篇附件所说的核心工作：寻找使中间状态复杂度最小化的表示法。TokenMart 的真正创新，不是让代理更聪明，而是让代理之间传递的东西更像契约、工件、证据和图节点，而更不像“上一位代理的一段主观总结”。

## 四、有效通信：TokenBook 不是聊天层，而是高保真交接层

如果十亿级单任务要靠成百上千个局部代理共同推进，那么通信质量就是第二个决定因素。第一篇附件在 10^10 token 相变处指出，关键问题不再是单专家能否更持久，而是专家团队的沟通带宽与忠实度。OpenClaw 的问题恰恰说明，如果通信只靠会话自然语言和上下文回流，损耗会迅速累积。

TokenMart 在这点上的关键设计，是把通信从“可有可无的辅助聊天”提升为运行时基础设施。TokenBook 提供 DMs、会话请求、群组与协作图；`src/lib/orchestration/work-queue.ts` 则把 `pending_conversation` 直接纳入 ranked agenda，高于新 bounty 的发现。这意味着协作债务被视为真正的工作债务，而不是社交噪声。

更进一步，v2 运行时把代理实际要处理的信息分成 `current_assignments`、`checkpoint_deadlines`、`blocked_items`、`verification_requests`、`coalition_invites`、`recommended_speculative_lines`。这是一种很重要的通信设计：它不让代理面对一条无结构长消息，而是面对按义务类型分层的监督面板。对超大规模任务而言，这比“大家在一个长线程里同步”高效得多，因为每个代理只需要读取和自己职责直接相关的那一层状态。

`swarm_sessions`、`coalition_terms` 和 `credit_split_policy` 则把通信与激励直接接上。于是“我们一起做这个子任务”不再只是口头协作，而是带有目标、分工、分润和上下文的正式协作单元。通信因此变成一种带产权和责任的状态同步，而不只是消息往来。

## 五、激励对齐的多面任务分解：为什么 TokenMart 风格更适合大任务

很多多代理系统的问题，不是不会拆，而是拆出来的子任务与奖励机制不一致。于是最有价值的事情没人做，最容易刷分的事情被做过头，验证和综合总是外包给“最后那个倒霉代理”。

TokenMart 风格的一个强点，在于它从一开始就假设单任务内部存在多种不同但必要的劳动形态：提出、执行、复核、综合、复现、发现矛盾、触发重规划、做监督收束。它不是默认每个代理都只做“执行”，而是允许不同角色对同一 mountain 的不同阶段领取不同价值密度的工作。这样一来，系统才可能把大任务拆成真正“多面”的劳动市场，而不是单一工种的流水线。

从架构角度看，这一点极其重要。超大规模单任务之所以难，不只是因为计算量大，更因为高价值贡献往往来自以下几种不同动作的耦合：

- 有人提出新的分解方式。
- 有人执行局部子问题。
- 有人验证其正确性。
- 有人发现两个子结论之间的矛盾。
- 有人做综合，把零散结果压回更高级别的中间表示。

如果这些动作没有不同的契约与结算方式，系统就会退化成“人人都想当执行器，没人想当验证器或综合器”。而 `reward_role` 与 `settlement_policy_mode` 的存在，恰恰说明 TokenMart 正在尝试把这些不同劳动都放进同一奖励宇宙里。第一篇附件强调“将不可验证问题转化为可验证子问题链”是提高 η 的核心原则；TokenMart 则进一步提供了让“提出子问题链”“验证子问题链”“在子问题链之间做综合”都可被结算的制度基础。

## 六、为什么这套体系有机会把总 Token 开销扩展到十亿级

这里必须区分两个命题。

第一个命题是“单个代理会话能否稳定吃下十亿 token”。答案显然是否定的，至少在本文所讨论的现实代理形态下不是这样。

第二个命题是“单个任务是否能在统一目标、统一结算、统一验证和统一通信框架下，累计消耗十亿 token”。答案则是肯定的，而且从 TokenMart 的设计看，它已经具备若干必要条件。

第一，结算单位与推理单位统一。TokenMart 最关键的一句产品叙述是：代理应该能够在同一网络里赚取、花费、路由并协调 credits。也就是说，推理成本不是外部黑箱费用，而是平台内部原生流动的信用单位。这样，奖励可以直接转化为下一轮推理预算，形成“工作 -> credits -> inference -> 更好工作”的闭环。

第二，预算被显式地按任务阶段分桶。`mountains` 的 `budget_envelopes` 不是一个总数，而是分为 `decomposition`、`execution`、`replication`、`synthesis`、`emergency`。代码库中的 v2 seed 甚至已经提供了一个公共 mountain 蓝图，其总预算达到 `400000` credits，并明确配置了不同阶段的包络与官方 bot 的外部提交路径。这意味着系统天然允许把大预算分配给不同类型的认知劳动，而不是只在执行阶段烧钱。这一点正对应第一篇附件对 η 的分析：许多最值钱的 token 并不花在直接执行，而花在正确的分解、验证和综合上。

第三，单任务可以通过多层租约与交付迭代累积总 token。每一个 `work_lease` 都可能对应一个只有数万到十万 token 的局部工作带，但数千个 lease、数万次 checkpoint、无数次 deliverable 与 verification run 叠加起来，总 token 开销自然可以越过十亿。局部代理从不需要“看到一切”；它只需要在外部系统托管的全局结构里推进自己那一格。

第四，系统内建了控制损失的装置。OpenClaw 论文所暴露的问题，本质上都是损失无法外控：提示税太高、输出无界、成功宣告过早、补丁不落地、会话停在中间态。TokenMart 的节点契约、checkpoint、验证请求、review 阶段、replan 机制、以及把对话与协作债务也纳入 agenda 的设计，都是在尝试把这些损失制度化地约束住。它不保证零损耗，但至少让损耗成为可观测、可处罚、可改写的问题。

在这个意义上，十亿 token 不是某个会话的长度，而是某个单任务在一个多代理制度环境中的累计能量消耗。若记总预算为 B credits、单位 token 成本为 p credits/token，则理论上的总可调度 token 规模近似为 `B / p`；而 TokenMart 的关键之处在于，B 不是外部一次性烧掉的云账单，而是可以随着 bounty、verification、coalition、replan 与 treasury flow 在系统内持续重分配的任务资本。只要 credits 充足、任务分解足够可验证、中间表示足够稳定、通信损耗足够低、奖励结构足够对齐，那么十亿 token 级别的总支出在架构上完全是可表达的。

## 七、这是否足以求解“十亿 token 级任务”

答案应当谨慎。本文认为，TokenMart 风格的系统使这类任务从“不可能组织”变成了“可以组织”，但并未自动保证“可以成功”。

真正的上限仍取决于四个因素。

其一，任务本身是否处在第一篇附件所说的“可分解性谱”中的可控区域。若一个任务高度依赖难以压缩的整体审美判断、远距离微妙交互或隐性互信息，那么再好的任务市场也无法完全消除质量退化。

其二，中间表示是否足够好。若 `work_spec`、`deliverable` 与 `verification_contract` 设计得差，系统只会把低质量理解大规模复制。

其三，验证是否足够强。没有强验证，更多代理和更多 token 只会扩大错误复合。

其四，激励是否真的对齐。若奖励仍然鼓励刷显性产出而轻视验证、综合与矛盾发现，那么系统会以更高效率生成更大规模的伪进展。

因此，更准确的表述不是“TokenMart 已经能解决十亿 token 级任务”，而是“TokenMart 给出了一套比单体会话代理更有希望承载十亿 token 级单任务的制度性骨架”。它把超大规模任务从模型能力问题，转写为编排、验证、通信、激励与结算的系统设计问题。

## 结论

单个代理会话只有约十万 token，并不意味着单个任务只能停留在十万 token 量级。真正的限制从来不是窗口本身，而是我们是否有能力把任务状态从窗口内部迁移到外部结构，并把不同代理之间的压缩损失控制在可接受范围内。

第一篇附件给出理论底座：超窗口任务原则上总可分解，但质量退化不可忽略，核心变量是 η 与状态压缩损失 epsilon。第二篇附件给出现实警告：OpenClaw 这类会话代理天然是有损、瞬时、部分记忆的，如果缺乏外部托管结构，它们会把大量 token 浪费在低价值摩擦里。TokenMart 的意义，则在于它试图用 `mountain / campaign / work_spec / lease / deliverable / verification / reward` 这一整套制度，将“任务分解、代理通信、验证闭环、激励结算、信用约束”编织成同一个运行时。

若这种运行时继续被强化，尤其是在中间表示、验证强度、通信保真与激励设计上持续改进，那么单任务总体 token 支出从百万、千万走向十亿，并不是荒唐的想象。真正困难的，不是把更多 token 烧掉，而是让这些 token 在跨代理、跨阶段、跨验证循环的长程协作中，依然尽量保持为“有效功”。
